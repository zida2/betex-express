[3] ATOMIC CAPACITY MANAGEMENT (Prevent Overbooking)
─────────────────────────────────────────────────────
Problem (Race Condition):
  • Scenario: Time slot capacity = 50, currently booked = 48 (2 slots left)
  • Client A clicks → Reads: capacity available (48 < 50) ✓
  • Client B clicks → Reads: capacity available (48 < 50) ✓
  • Client A confirms → Backend: booked = 48 + 1 = 49 ✓
  • Client B confirms → Backend: booked = 48 + 1 = 49 ❌ (Should be 50!)
  • RESULT: Overbooking! 51 clients assigned to capacity 50

Impact:
  • Time slot exceeds physical capacity
  • Operational nightmare for delivery team
  • Some clients can't be served properly

Current Status:
  • UI shows: booked: 42, capacity: 50 (demo data)
  • No atomic operations → Race condition possible
  • Multiple concurrent clients can overbook the slot

Required Implementation (Database Transaction):
  ✓ Use atomic SQL: UPDATE time_slots SET booked = booked + 1 
                    WHERE id = ? AND booked < capacity
  ✓ If UPDATE returns 0 rows affected → Slot is full (reject)
  ✓ If UPDATE returns 1 row affected → Success (slot reserved)
  ✓ This ensures only ONE INCREMENT per client, never overbooking

Affected Files:
  • backend/src/services/deliveryRequestService.js (atomic check/increment)
  • backend/src/models/TimeSlot.js (database configuration)
  • database migration: Ensure transaction support

Requirement References:
  - Requirement 11 AC 4: Atomic updates to prevent overbooking
  - Requirement 4 AC 5: Decrement available slot capacity atomically

[5] PRICE LOCKING AFTER CONFIRMATION
──────────────────────────────────────
Problem:
  • Current: Price calculated but never locked
  • Should Be: After client clicks "Confirm", price is frozen
  • Impact: Medium (affects billing, creates disputes if price changes)

Scenario:
  • Client selects driver at 2.5km → Shows: 1125 FCFA
  • Client reviews and clicks "Confirm"
  • Driver location updates → Distance now 3.1km
  • System might recalculate → 1275 FCFA
  • Client charged 1275 FCFA but expected 1125 FCFA ❌ BILLING DISPUTE

Required Implementation:
  ✓ Add fields to database: confirmed_price, price_locked_at
  ✓ When client confirms: Lock price (set confirmed_price = current price)
  ✓ Never update confirmed_price after lock (validation rule)
  ✓ Use confirmed_price for billing (not estimated_price)
  ✓ UI indicator: Show "✓ Price locked: 1125 FCFA"

Affected Files:
  • backend/src/models/DeliveryRequest.js (add lock fields)
  • backend/src/services/deliveryRequestService.js (locking logic)
  • frontend/src/components/DeliveryOptions.js (UI indicator)
  • database migration: Add confirmed_price, price_locked_at fields

Requirement References:
  - Requirement 6 AC 5: Lock price after confirmation
  - Requirement 8 AC 3: Maintain original pricing (don't recalculate)
